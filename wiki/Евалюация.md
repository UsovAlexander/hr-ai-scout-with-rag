# Евалюация

Оффлайн-оценка качества retrieval — **ключевая часть проекта для портфолио**.
Превращает «ещё один RAG-туториал» в «RAG-систему с доказанным качеством».
Связано с [[Датасет]] (ground truth), [[Hybrid_Search]] (что оцениваем),
[[Векторная_БД]] (ablation моделей).

## Идея

Существующая разметка `target` используется как **ground truth**. Для каждой
вакансии, у которой есть ≥1 резюме с `target = 1`:
1. прогнать `search_resumes(vacancy_id, top_k=...)`,
2. посмотреть, на каких позициях оказались истинно релевантные (`target = 1`)
   резюме,
3. посчитать метрики и усреднить по вакансиям.

Релевантность бинарна: `target = 1` → релевантно, `target = 0` → нет.

## Метрики (считаем сами, без готовых библиотек)

| Метрика | Что показывает | Как считаем |
|---|---|---|
| `Recall@10` | Сколько `target=1` резюме попало в топ-10 | TP@10 / всего `target=1` для вакансии |
| `Precision@10` | Какая доля топ-10 реально релевантна | TP@10 / 10 |
| `MRR` | Насколько высоко первое релевантное резюме | 1 / rank первого `target=1` |
| `NDCG@10` | Качество ранжирования с учётом позиции | стандартная формула с log-дисконтом |

Метрики реализуются вручную (`eval/retrieval_metrics.py`) — чтобы явно
показать понимание формул (ограничение из стека: не использовать готовую
RAG-eval библиотеку).

## Главный эксперимент

Сравнить три режима поиска из [[Hybrid_Search]]:
**dense-only / BM25-only / hybrid+RRF** — таблица в
`eval/results/comparison.md`.

Опционально (если останется время): сравнить две модели эмбеддингов из
[[Векторная_БД]] (ablation через конфиг).

## Бейзлайны для сравнения

- случайный ранкинг,
- и/или существующий classification score (вероятность `target` из CatBoost,
  если он есть как отдельная модель) — чтобы показать, что RAG-поиск
  конкурентен или превосходит по recall кандидатов, которых классификатор
  даже не рассматривал бы.


## Реализовано (Этап 4)

Код: `eval/retrieval_metrics.py` (метрики с нуля, self-test),
`eval/build_eval_set.py` (сэмплинг ground truth), `eval/run_eval.py`
(оркестрация + `eval/results/comparison.md`).

Настройки прогона: 500 вакансий (seed=42), `k=10`, глубина выдачи 100,
relevance бинарная по `target=1`, модель `LaBSE-en-ru`. По датасету: все 3 409
вакансий имеют ≥1 позитив; в среднем 7.7 релевантных и 97.7 размеченных
кандидатов на вакансию.

### Результаты (macro-average по 500 вакансиям)

| mode | recall@10 | precision@10 | mrr | ndcg@10 |
|---|---|---|---|---|
| random | 0.0006 | 0.0004 | 0.0014 | 0.0004 |
| **bm25** | **0.4349** | **0.3128** | **0.7480** | **0.4643** |
| dense | 0.1120 | 0.0772 | 0.2737 | 0.1151 |
| hybrid | 0.2427 | 0.1732 | 0.5494 | 0.2677 |

### Ключевой вывод (неожиданный и ценный для портфолио)

- **BM25 ≫ dense** на этих данных: решение рекрутера о шортлисте сильнее
  коррелирует с явным совпадением навыков/термов, чем с эмбеддинг-близостью по
  длинному склеенному тексту резюме.
- **Наивный hybrid (равновесный RRF) НЕ обгоняет BM25** — равные веса размывают
  сильный лексический сигнал. → мотивирует **взвешенный** fusion как следующий шаг.
- `random ≈ 0.0006` точно совпал с теоретическим полом (~k/|corpus|) —
  подтверждает корректность реализации метрик и джойна id.
- На отдельных вакансиях hybrid может выигрывать (спот-чек «SAP ABAP»:
  dense 1/8, bm25 2/8, **hybrid 3/8**) — проигрыш только в среднем.

Полный отчёт: `eval/results/comparison.md`. Per-vacancy метрики:
`eval/results/per_vacancy_metrics.parquet`.

## Open questions

- ~~Какой `k` отчитывать~~ — **решено (Этап 4):** метрики @10, MRR считается по
  глубине 100. Можно расширить до {5,10,20} при желании.
- ~~Стратегия сэмплинга~~ — **решено:** случайные 500 вакансий с ≥1 позитивом
  (seed=42), `--limit 0` для всех 3 409.
- **Precision@10 — нижняя оценка** (подтверждено количественно): поиск идёт по
  всем 20 845 резюме, а размечено лишь ~98 на вакансию, поэтому непомеченные в
  топ-10 считаются нерелевантными, хотя могут таковыми не быть. Recall@10/MRR/NDCG
  надёжнее. Из-за декартова произведения ([[Датасет]]) это структурный нюанс.
- Не сравнивали вторую модель эмбеддингов (e5) — опциональный ablation из спеки.
- Взвешенный RRF / тюнинг пула — не делали (см. [[Hybrid_Search]]).
