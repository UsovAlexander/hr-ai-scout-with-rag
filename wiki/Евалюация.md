# Евалюация

Оффлайн-оценка качества retrieval — **ключевая часть проекта для портфолио**.
Превращает «ещё один RAG-туториал» в «RAG-систему с доказанным качеством».
Связано с [[Датасет]] (ground truth), [[Hybrid_Search]] (что оцениваем),
[[Векторная_БД]] (ablation моделей).

## Идея

Существующая разметка `target` используется как **ground truth**. Для каждой
вакансии, у которой есть ≥1 резюме с `target = 1`:
1. прогнать `search_resumes(vacancy_id, top_k=...)`,
2. посмотреть, на каких позициях оказались истинно релевантные (`target = 1`)
   резюме,
3. посчитать метрики и усреднить по вакансиям.

Релевантность бинарна: `target = 1` → релевантно, `target = 0` → нет.

## Метрики (считаем сами, без готовых библиотек)

| Метрика | Что показывает | Как считаем |
|---|---|---|
| `Recall@10` | Сколько `target=1` резюме попало в топ-10 | TP@10 / всего `target=1` для вакансии |
| `Precision@10` | Какая доля топ-10 реально релевантна | TP@10 / 10 |
| `MRR` | Насколько высоко первое релевантное резюме | 1 / rank первого `target=1` |
| `NDCG@10` | Качество ранжирования с учётом позиции | стандартная формула с log-дисконтом |

Метрики реализуются вручную (`eval/retrieval_metrics.py`) — чтобы явно
показать понимание формул (ограничение из стека: не использовать готовую
RAG-eval библиотеку).

## Главный эксперимент

Сравнить три режима поиска из [[Hybrid_Search]]:
**dense-only / BM25-only / hybrid+RRF** — таблица в
`eval/results/comparison.md`.

Опционально (если останется время): сравнить две модели эмбеддингов из
[[Векторная_БД]] (ablation через конфиг).

## Бейзлайны для сравнения

- случайный ранкинг,
- и/или существующий classification score (вероятность `target` из CatBoost,
  если он есть как отдельная модель) — чтобы показать, что RAG-поиск
  конкурентен или превосходит по recall кандидатов, которых классификатор
  даже не рассматривал бы.

## Файлы (Этап 4)

```
eval/
├── build_eval_set.py     # сэмплинг из target=1/0 для оценки
├── retrieval_metrics.py  # NDCG@k, MRR, Recall@k, Precision@k
└── results/              # comparison.md и пр.
```

Результаты показываются и в Streamlit UI (вкладка «Метрики качества») —
живая демонстрация, а не текст в README ([[Roadmap]] Этап 6).

## Open questions

- `k` фиксировано на 10 в таблице метрик, но интерфейс `search_resumes`
  использует `top_k=20` по умолчанию — какой набор `k` отчитывать (только 10
  или {5,10,20})? Не зафиксировано.
- Стратегия сэмплинга в `build_eval_set.py` (сколько вакансий, как
  балансировать) не описана детально.
- Из-за декартова произведения ([[Датасет]]) «не-target=1» резюме для вакансии
  не обязательно нерелевантны — они просто не размечены как `target=1`.
  Трактовка непомеченных как негативов может занижать precision. Отметить как
  методологический нюанс при интерпретации.
